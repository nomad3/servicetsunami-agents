# apps/memory-core

Rust gRPC service implementing the memory-first hot path: `Recall`, `RecordObservation`, `RecordCommitment`, `IngestEvents`. Talks to PostgreSQL via pgvector (768-dim) and to the embedding service for vectorization.

## Wiring

- Listens on **gRPC port 50052**.
- Service contract: [`proto/memory.proto`](proto/memory.proto). Stubs generated by `build.rs` via `tonic-build`.
- Dependencies:
  - PostgreSQL 13 + pgvector (talks via `sqlx`).
  - [`../embedding-service/`](../embedding-service/) on port 50051 for generating embeddings.
- API client: `apps/api/app/memory/recall.py`, `record.py`, `ingest.py`. Phase 2 dual-read enabled — Python primary, Rust shadow comparison for cutover validation.

## What it does

| RPC | Purpose |
|-----|---------|
| `Recall(query, tenant, limit, token_budget)` | pgvector semantic search across entities, observations, episodes, commitments. Returns ranked `EntitySummary` with embeddings, decay, and confidence. 1500ms hard cap. |
| `RecordObservation` | sync write of a fact / insight, embedded inline. |
| `RecordCommitment` | sync write of a goal / commitment with optional due date. |
| `IngestEvents` | bulk write from source adapters (email, calendar, chat, browser activity). |

| `RUST_LOG` | `memory_core=info` | tracing |

## Hard rules

- **Tenant isolation in every query.** The `tenant_id` is part of every RPC; the SQL must filter on it. Same invariant as the API service — multi-tenant break otherwise.
- **No business logic here.** This service is the pgvector + embedding hot path; policy decisions (which tier of memory to prefer, how to mix entities and observations) live in the API's `memory/` package.
- **Phase 2 dual-read** — Python primary, Rust shadow. Don't break the contract; the shadow comparison runs on every recall and any divergence shows up in metrics.
