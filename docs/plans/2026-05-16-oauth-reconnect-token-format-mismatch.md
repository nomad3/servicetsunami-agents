# OAuth reconnect token-format mismatch — root cause analysis

**Date:** 2026-05-16
**Scope:** Claude Code + Gemini CLI subprocess auth fails with `401 Invalid bearer token` after a successful OAuth reconnect from the Integrations UI.
**Affected users:**
- **saguilera** (tenant `752626d9-8b2c-4aa2-87ef-c458d48bd38a`) — Claude Code reconnected at 2026-05-16 21:11:53 UTC (DB `updated_at` on `session_token` row).
- **James / jacktanners26@gmail.com** (tenant `74003832-dc9c-4ce5-9ab6-6bf402f79ecb`, "jack something") — Gemini CLI reconnected at 2026-05-16 21:01:20 UTC.

This is a **diagnosis-only** document. No code is changed.

---

## TL;DR

**Hypothesis #1 is correct for Claude Code.** The `/claude-auth` UI flow runs `claude auth login --claudeai`, which is the **interactive subscription login** — not `claude setup-token`. Per the Anthropic docs (`code.claude.com/docs/en/iam`):

> `CLAUDE_CODE_OAUTH_TOKEN` environment variable. **A long-lived OAuth token generated by `claude setup-token`.** Use this for CI pipelines and scripts where browser login isn't available.

> The command [`claude setup-token`] walks you through OAuth authorization and **prints a token to the terminal. It does not save the token anywhere**; copy it and set it as the `CLAUDE_CODE_OAUTH_TOKEN` environment variable wherever you want to authenticate.

The `auth login --claudeai` flow writes interactive subscription credentials to `~/.claude/.credentials.json` (or the encrypted macOS Keychain). Those are **not** in the format `CLAUDE_CODE_OAUTH_TOKEN` accepts — they are session credentials managed by `/login` and `/logout`, refreshed in-process by the CLI's own runtime, and never intended to be hoisted out via env var.

`apps/api/app/api/v1/claude_auth.py:_persist_credentials` (line 601) then performs a salvage glob over `CLAUDE_CONFIG_DIR`, grabbing whatever string-shaped field it finds first under the keys `oauth_token / accessToken / access_token / token`, falling back to the raw JSON blob, and stores that as `credential_key='session_token'` in the vault. The code-worker pulls that string out of the vault unchanged and exports it as `CLAUDE_CODE_OAUTH_TOKEN`. Anthropic's API rejects it because it's a *session* artefact, not a *long-lived setup-token* artefact — different scopes, different audiences, different shape.

**Hypothesis #3 (same pattern for Gemini) is correct in shape but the failure mechanism is different.** Gemini's flow does write the full `oauth_creds.json` blob the CLI itself produced, mirrors the `access_token` + `refresh_token` to separate vault keys, and the executor materialises a `GEMINI_HOME/.gemini/oauth_creds.json` from that blob. The bytes round-trip cleanly. The 401 surface for James is therefore an **expired access_token without an in-process refresh on the code-worker side** — the vault rows for tenant `74003832…` were last `updated_at` 2026-04-22 (saguilera Gemini side: 2026-04-22). The reconnect at 21:01 today only refreshed the `updated_at` (vault re-stamping) — it did **not** in fact re-run the `gemini` OAuth flow, OR the gemini-cli's first call refreshed the token in its own ephemeral GEMINI_HOME and the new token was never written back to the vault because the home is a per-task sandbox (`apps/code-worker/workflows.py:_prepare_gemini_home`). Either way the symptom is the same: a stale access_token gets handed to Google and rejected.

**Hypothesis #5 (PR #531 exposed the bug) is confirmed.** PR #531 (`43c5b3b6`, merged 2026-05-16 17:06 UTC) added `env.pop("ANTHROPIC_API_KEY", None)` on the OAuth branch of `cli_executors/claude.py`. Before that, the inherited Anthropic Console API key from `apps/api/.env` ranked **above** `CLAUDE_CODE_OAUTH_TOKEN` in Claude Code's auth precedence (item 3 vs item 5 in the docs' `Authentication precedence` list), so subscription users were silently routed to the Console billing path. The user-facing complaint was "Credit balance is too low", not "Invalid bearer token", because the API key was being used, not the bogus OAuth token. Today's drop of `ANTHROPIC_API_KEY` revealed the underlying token-format bug for the first time on production traffic.

**Hypothesis #2 (refresh token never used) is partially correct but downstream of #1.** The OAuth flow does NOT store a refresh_token for `claude_code` at all (verified: vault only has `credential_key='session_token'` rows for tenant 752626d9…, no `refresh_token` row). Even if the executor read a refresh_token, there is no Anthropic OAuth refresh endpoint we are calling; the CLI's own subscription credential lifecycle is opaque to us.

**Hypothesis #4 (`auth_json` vs raw token) is the right design pattern but only currently applied to Codex.** `workflows.py:1187` (`raw_auth = creds.get("auth_json") or creds.get("session_token")`) is the Codex executor and works correctly because `codex_auth.py:_persist_auth_json` stores the **entire `~/.codex/auth.json` blob** under `credential_key='auth_json'`, then `_prepare_codex_home` writes it back to a per-session `CODEX_HOME` so the codex CLI reads its own auth file on disk exactly as it wrote it. That's the correct model for any CLI whose auth is a stateful local file rather than a single bearer header. Claude Code does not currently follow this pattern.

Note on saguilera: the `oauth_creds/oauth_token/refresh_token` triple on the same tenant belongs to a different `integration_config_id` (gemini_cli). The Claude-side row is only `session_token` — there is no `refresh_token` row for Claude. This is consistent with `_persist_credentials` in `claude_auth.py` writing exactly one vault key (`session_token`) regardless of what was in `CLAUDE_CONFIG_DIR`.

The freshly stored saguilera `session_token` row (21:11:53) is unambiguously today's reconnect (the 21:15 `updated_at` is a touch from `retrieve_credentials_for_skill` writing `last_used_at`).

### 2. What `_persist_credentials` actually stores (`apps/api/app/api/v1/claude_auth.py:601`)

The function globs every `*.json` under the per-tenant `CLAUDE_CONFIG_DIR` and picks the first string-shaped field matching `oauth_token / accessToken / access_token / token` (lines 623–631). If none match it dumps the whole dict back to a string (line 634) or even concatenates every JSON file in the dir (line 668). The vault then receives whatever bytes won that priority race — usually `.credentials.json`'s `accessToken` field — and stores them under `credential_key='session_token'`, `credential_type='oauth_token'`.

That field is the interactive subscription's session credential, which the CLI's own runtime knows how to refresh via Anthropic's internal endpoints. It is **not** the `sk-ant-oat01-…` long-lived OAuth token shape that `claude -p` accepts via `CLAUDE_CODE_OAUTH_TOKEN`.

### 3. The CLI emits two different artefacts depending on which command was run

- `claude auth login --claudeai` (what we run): interactive flow. Writes `.credentials.json` under `CLAUDE_CONFIG_DIR`. The token inside is a session credential.
- `claude setup-token` (what `CLAUDE_CODE_OAUTH_TOKEN` requires): prints a long-lived OAuth token to stdout, **does not save it anywhere**.

The commit `c54f91b3` (2026-04-05) is the original `claude_auth.py` addition, with the explicit message:

> Replaces deprecated `setup-token` flow with proper OAuth

This is the inversion point. `setup-token` is not deprecated — it is the documented path for headless use. The "proper OAuth" interactive flow that replaced it produces a credential the rest of our system cannot use.

We have `claude setup-token` available in both the `api` and `code-worker` containers (verified: `claude --version` reports 2.1.143, `claude setup-token --help` returns the long-lived token doc).

### 4. PR #531 timeline confirms

```
43c5b3b6 fix(claude): drop inherited ANTHROPIC_API_KEY when using OAuth subscription (#531)
         2026-05-16 17:06:04 UTC
```

Before PR #531:
- `cli_executors/claude.py` set `CLAUDE_CODE_OAUTH_TOKEN` (broken token), but the inherited container-env `ANTHROPIC_API_KEY` ranked higher in Claude Code's auth precedence, so the CLI silently fell back to Console API billing. Users saw "Credit balance is too low", not "Invalid bearer token".
- The bug was masked.

After PR #531:
- `env.pop("ANTHROPIC_API_KEY", None)` strips the fallback.
- Now `CLAUDE_CODE_OAUTH_TOKEN` is the only header presented to Anthropic.
- Anthropic rejects it as malformed → `401 Invalid bearer token`.

saguilera's chat **was working before PR #531 deployed today** — but only because every turn was being billed to the Console API key, not to her subscription. The "fix" exposed the deeper bug.

### 5. Gemini comparison: same shape, different mechanism

`apps/api/app/api/v1/gemini_cli_auth.py:_persist_creds` (line 287) does the right thing for Gemini:

1. Reads the **full** `oauth_creds.json` that gemini-cli wrote, stores it verbatim under `credential_key='oauth_creds'` (line 319).
2. Also extracts `access_token` → `credential_key='oauth_token'` and `refresh_token` → `credential_key='refresh_token'` for convenience.

The executor (`cli_executors/gemini.py:69`) then reassembles an `auth_payload` dict from those vault rows and passes it to `_prepare_gemini_home` (`workflows.py:1569`), which writes a fresh `.gemini/oauth_creds.json` into a per-task sandbox directory. **Gemini CLI then reads that file on startup and refreshes the access_token in-process using the bundled refresh_token**, writing the rotated token back into the *sandbox* dir — never back into the vault. So the next code-worker task starts with the same stale access_token from the vault again.

For James:
- Vault `oauth_creds.json.expiry_date` (inside the blob) is from his original auth on 2026-05-05.
- Today's "reconnect" at 21:01 only re-stored the same blob (or possibly never actually re-ran the OAuth handshake — the manager state-machine in `gemini_cli_auth.py` returns the in-memory state if a flow is still `pending`/`starting`).
- The access_token inside the blob is expired (> 11 days old, Google access_tokens are 1h TTL).
- Gemini CLI tries to refresh on first call but if the refresh fails (e.g., user revoked the grant on Google's side, or the refresh_token itself expired) it 401s and the executor surfaces "Invalid bearer token".

So Gemini's bug is **not the same as Claude's**. Claude's vault content is structurally wrong; Gemini's vault content is structurally right but stale + never rotated back. The two are independent fixes.

## Fix shape

### Claude Code (structural fix — vault content is wrong)

**Option A (recommended): switch the UI flow to `claude setup-token`.**

`apps/api/app/api/v1/claude_auth.py:_run_login` currently spawns `claude auth login --claudeai` and forwards stdin for the paste-code. Replace with `claude setup-token` (or run `setup-token` as the second stage, after `auth login` has established the subscription session inside the per-tenant `CLAUDE_CONFIG_DIR`). `setup-token` prints the long-lived `sk-ant-oat01-…` token to stdout, which is exactly the shape `CLAUDE_CODE_OAUTH_TOKEN` needs. Capture stdout, strip ANSI, store the raw string under `credential_key='session_token'`, `credential_type='oauth_token'`.

Functions to change:
- `apps/api/app/api/v1/claude_auth.py:_run_login` (cmd spawn + stdout capture)
- `apps/api/app/api/v1/claude_auth.py:_persist_credentials` (replace the glob salvage with `state.captured_token` from the new stdout capture)
- No executor-side change required — the vault contract (`session_token` → `CLAUDE_CODE_OAUTH_TOKEN`) stays the same.

`setup-token` requires an interactive TTY for the OAuth flow (Ink raw mode — see commit `b232e522` from 2026-03-11). The current implementation already pipes stdin/stdout; the OAuth verification code (just like `auth login --claudeai`) is a short string that can be forwarded over the same `/submit-code` route. Confirm the TTY requirement still applies in claude CLI 2.1.143; if so, spawn under `pty.openpty()` like `gemini_cli_auth.py` already does.

**Option B (matches Codex pattern): store the full credentials.json blob, materialise it on every task.**

Stop using `CLAUDE_CODE_OAUTH_TOKEN`. Instead:
- `claude_auth.py` runs `auth login --claudeai`, then reads the resulting `.credentials.json` from `CLAUDE_CONFIG_DIR` and stores it verbatim under `credential_key='credentials_json'`.
- `cli_executors/claude.py` writes a per-task `CLAUDE_CONFIG_DIR/.credentials.json` from the vault blob, then invokes `claude -p` with `CLAUDE_CONFIG_DIR` pointing at the sandbox dir, no env tokens at all.
- Same caveat as Gemini: the CLI rotates the session credential in its own dir, and we'd need a write-back step to capture the rotated blob back into the vault to avoid the same staleness Gemini suffers. Optional, since the subscription session TTL is much longer than Google's 1h access_token.

This is more code than Option A but is closer to how the CLI is actually designed to be authenticated and doesn't depend on the `setup-token` shape staying stable.

**Migration concern:** Yes. Every existing `claude_code.session_token` row in the vault is the wrong shape and must be re-stored. Users must hit Connect again in the Integrations UI after the fix ships. The simplest path is a forced revoke of every active `claude_code.session_token` row in a follow-up migration so the UI shows "Disconnected" and prompts re-auth.

### Gemini CLI (refresh-rotation fix — vault content is right, just stale)

The executor must capture the rotated `oauth_creds.json` from the per-task `GEMINI_HOME` and write it back to the vault after a successful run. Alternatively, refresh the access_token explicitly *before* spawning gemini-cli (using the stored refresh_token + the known `681255809395-…` client_id against `https://oauth2.googleapis.com/token`) and write the fresh access_token into the materialised `oauth_creds.json` and the vault.

Functions to change:
- `apps/code-worker/cli_executors/gemini.py` — add post-run capture of `<gemini_home>/.gemini/oauth_creds.json`, compare timestamps, and POST the rotated blob back to a new `/oauth/internal/store/gemini_cli` API endpoint.
- `apps/api/app/api/v1/oauth.py` — add the internal write-back endpoint, gated by `X-Internal-Key`.
- Optional: a pre-spawn refresh helper in `workflows.py` that hits Google's token endpoint directly if the cached access_token's `expiry_date` is in the past.

**Migration concern:** No — once the refresh logic is in place, existing vault rows are fine as long as the `refresh_token` is still valid. Users whose refresh_token has *also* expired (Google's refresh_tokens expire after 6 months of disuse or on user revocation) will need to reconnect.

### Cross-cutting

- Both flows should grow a self-test step that, immediately after `_persist_*`, runs a minimal probe (`claude -p "hi"` or `gemini -p "hi"` with the just-stored credential, capped at 30s) and only marks `status='connected'` if the probe returns 200. Today the UI shows "connected" purely based on a vault row existing, which is how both saguilera and James got the optimistic-but-wrong green light.
