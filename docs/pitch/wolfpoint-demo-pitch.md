# Wolfpoint.ai — The Network Layer for AI Agents

## From Isolation to Civilization: Why AI Needs What Made Us Human

---

### Act I: The Lonely Human

200,000 years ago, Homo sapiens was just another primate. Stronger species existed. Faster ones too. We had no claws, no venom, no armor.

What we had was something invisible: **the ability to coordinate.**

A single human is fragile. But humans who could communicate, specialize, and trust each other became unstoppable. The hunter didn't need to be the toolmaker. The toolmaker didn't need to be the healer. Each excelled at one thing, and the network made them all greater than the sum.

This is the **sociability hypothesis** — the idea that human dominance came not from individual intelligence, but from our ability to form coordinated networks of specialists.

Language gave us shared context.
Trade gave us specialization.
Institutions gave us trust at scale.
Writing gave us memory across generations.

Every major leap in civilization — agriculture, cities, science, industry, the internet — was not a leap in individual capability. It was a leap in **coordination infrastructure.**

### The Five Breakthroughs

**1. Shared Tools, Not Shared Models**

Every agent — Claude, Codex, Gemini, local Gemma 4 — connects to the same 81 MCP tools. Email, calendar, Jira, GitHub, knowledge graph, analytics. Connect once, available to all. Like roads connecting villages — they don't care who's traveling on them.

**2. Shared Memory That Survives Sessions**

The Knowledge Graph persists across conversations, agents, and platforms. Claude creates a contact entity. Codex finds it next week. Gemma 4 recalls it instantly. 337 entities, 4,823 observations, all semantically searchable.

This is writing for AI — memory that outlasts any single conversation.

**3. Multi-Provider Review Council**

No single model is trusted blindly. When a response matters, Claude, Codex, and Gemma 4 each independently review it. Agreement = high confidence. Disagreement = human escalation.

```
Response: "The contract value is $2.5M"

Claude Review:  APPROVED (score: 88) — "Factually correct"
Codex Review:   APPROVED (score: 99) — "Accurate and concise"
Gemma 4 Review:    APPROVED (score: 95) — "Correctly sourced"

Agreement: 1.0 | Consensus: PASSED | Recommended: Codex
```

This is peer review for AI — the same mechanism that made science trustworthy.

**4. Reinforcement Learning That Routes by Performance**

Every response is scored. Every platform's performance is tracked. The system learns:

| Task Type | Best Platform | Avg Score | Experiences |
|-----------|--------------|-----------|-------------|
| General chat | Claude Code | 0.594 | 2,610 |
| Code tasks | Claude Code | 0.800 | 15 |
| Data queries | *collecting...* | — | — |

Over time, routing becomes automatic. The coordinator doesn't guess — it knows. Like a society that has figured out which specialist to call for each job.

**5. Zero Lock-In**

Switch providers in seconds. Each tenant brings their own subscriptions — Claude Pro, ChatGPT Plus, Gemini Pro. No API credits. No vendor lock-in. When a better model launches, add it to the network. The review council will tell you if it's actually better.

### Deep Dive: The Review Council (How AI Peer Review Works)

The multi-provider review council is the mechanism that makes the network trustworthy. Without it, you're just routing to models blindly.

**Architecture:**

```
Agent Response (already returned to user — zero latency added)
  │
  ▼
Auto Quality Scorer (local Gemma 4, always-on, free)
  ├── 6-dimension rubric (accuracy, helpfulness, tool usage, memory, efficiency, context)
  ├── 3-reviewer consensus (Accuracy, Helpfulness, Persona reviewers)
  ├── Leave-one-out fragility detection
  │
  └── Decision gate: Should we get a second opinion?
        │
        ├── Side-effect tools used (email, deploy) → YES
        ├── Fragile consensus (2/3 exactly) → YES
        ├── Low score (<40) → YES
        ├── Random 5% sample → YES
        │
        ▼
  Multi-Provider Review Council (async Temporal workflow)
  ├── Claude reviews (using tenant's subscription, sonnet model)
  ├── Codex reviews (using tenant's subscription)
  ├── Gemma 4 reviews (local, always free)
  │
  ▼
  Meta-Adjudicator
  ├── Agreement score (0.0 to 1.0 across ALL reviewers)
  ├── Disagreement detection (which providers disagree and why)
  ├── Platform recommendation (who scored highest)
  └── Merged into RL experience for routing optimization
```

**Key design decisions:**
- Reviews are async — the user already has their response. Zero latency impact.
- Each provider reviews independently — no knowledge of other reviewers' verdicts.
- Failed providers are counted as non-approvals — no false "100% agreement" when 2 of 3 crash.
- The council only runs when it matters — trivial messages are skipped.

### Deep Dive: Learned Routing (How AI Societies Self-Organize)

The learned routing system is what transforms a static multi-model setup into a self-improving network. It answers the question: **"For this specific task, which specialist should I call?"**

**The Modularity Hypothesis Applied:**

Just as human societies learned that blacksmiths make better swords than farmers, Wolfpoint learns that Claude writes better code than Codex, but Codex answers factual questions faster.

```
Message arrives
  │
  ▼
Agent Router (< 10ms, no model calls)
  │
  ├── Keyword matching: infer task type (code, data, sales, general)
  │
  ├── RL Platform Lookup: query historical performance
  │     SELECT platform, AVG(reward)
  │     FROM rl_experiences
  │     WHERE task_type = 'code'
  │     GROUP BY platform
  │     → Result: claude_code avg=0.800 over 15 experiences
  │
  ├── Confidence check: enough data? clear winner?
  │     → confidence = min(1.0, experiences/50) × gap_factor
  │     → If confidence ≥ 0.4: override default platform
  │
  └── Exploration mode (training phase):
        → Route % of traffic to underexplored platforms
        → Currently: 90% Codex, 10% Claude (building Codex dataset)
```

**The feedback loop:**

```
Route → Respond → Score → Learn → Route Better
  │                                      │
  └──────────────────────────────────────┘
```

Every response generates an RL experience. Every RL experience makes the next routing decision slightly better. Over thousands of interactions, the system converges on the optimal specialist for each task type.

**Training strategy:**
1. **Explore** — deliberately route to underused platforms (current phase)
2. **Compare** — provider council gives cross-platform quality signals
3. **Exploit** — RL system routes to the proven winner per task type
4. **Monitor** — provider council detects regressions from model updates

This is the same explore-exploit tradeoff that drives all reinforcement learning — and all human skill acquisition.
